## 为什么

当前项目尚未形成可实施的家谱系统方案。我们需要一个低运维、易备份、可逐步扩展的前后端架构，支持家谱数据管理、可视化展示与多人协作权限控制。现在明确方案可以避免后续实现阶段反复返工。

## 变更内容

- 定义家谱系统的核心产品边界：人物、关系、事件、资料附件的管理与可视化。
- 采用 `一个家谱 = 一个 SQLite 文件` 的数据隔离模型。
- 定义应用启动时扫描家谱目录并注册可用家谱文件的机制。
- 定义家谱级 RBAC（`creator`、`admin`、`member`）及关键权限边界。
- 定义后端按 `family_id` 路由到目标 SQLite 文件的服务模式。
- 定义关键审计、迁移、容错与并发约束（WAL、busy_timeout、损坏文件跳过）。

## 功能 (Capabilities)

### 新增功能
- `family-file-registry`: 启动扫描家谱目录、校验并注册单家谱 SQLite 文件，提供可用家谱清单与状态。
- `genealogy-data-management`: 定义人物、关系、事件、附件等核心数据模型与读写规则，支持按家谱隔离。
- `family-rbac`: 定义家谱级角色模型（creator/admin/member）与权限判定、角色变更约束及审计要求。
- `tree-query-and-visualization`: 定义用于前端树图/关系图展示的查询接口契约与分页/按层加载规则。

### 修改功能
- （无）

## 影响

- 影响后端 API 设计：所有业务接口需显式包含 `family_id`。
- 影响存储策略：从单库思路转为“多文件、按家谱隔离”的 SQLite 管理模式。
- 影响权限体系：引入家谱级角色与审计约束。
- 影响前端信息架构：新增家谱列表、家谱切换、树图数据加载策略。
- 影响部署与运维：需要家谱目录扫描、单库迁移、备份与异常库告警机制。
