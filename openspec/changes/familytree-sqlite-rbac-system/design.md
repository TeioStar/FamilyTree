## 上下文

当前仓库尚未落地业务代码，但已完成目标澄清：构建一个前后端家谱管理与可视化系统。系统需要支持多人协作、角色权限、长期数据积累与低运维部署。

关键约束如下：
- 数据隔离：一个家谱必须对应一个独立 SQLite 文件。
- 启动发现：应用启动时必须扫描家谱目录并注册可用家谱。
- 权限边界：采用家谱级三角色（creator/admin/member）。
- 可视化需求：前端需要按层加载家谱树，避免大树一次性渲染导致卡顿。
- 可靠性要求：损坏家谱文件不能拖垮整体服务，必须可降级。

相关利益相关方：
- 家谱创建者（关注数据主权、可恢复性、可管理性）
- 家谱管理员（关注成员协作与权限配置）
- 家谱成员（关注录入效率与可视化浏览）
- 维护者（关注部署简单、排障可观测）

## 目标 / 非目标

**目标：**
- 定义单家谱单 SQLite 文件的系统架构与运行流程。
- 定义启动扫描、注册、校验、迁移、异常隔离机制。
- 定义人物/关系/事件/附件的基础数据契约与读写边界。
- 定义 creator/admin/member 的权限矩阵及硬约束。
- 定义供前端树图与关系图调用的查询契约。
- 定义关键操作审计和回滚准备机制。

**非目标：**
- 不在本变更中实现具体 UI 视觉稿和交互细节。
- 不在本变更中引入 Neo4j、ES 等额外存储系统。
- 不在本变更中实现复杂实时协作（如 CRDT）。
- 不在本变更中覆盖跨家谱合并/联邦查询。

## 决策

### 决策 1：采用“一个家谱一个 SQLite 文件”
- 结论：每个家谱使用独立 `.ftree.db` 文件，统一存放于家谱目录。
- 原因：
  - 天然隔离，降低跨家谱数据污染风险。
  - 文件级备份和迁移简单，便于家族私有化保管。
  - 适合中小规模协作场景，无需复杂基础设施。
- 备选方案：
  - 单 PostgreSQL 多租户：扩展性高，但运维复杂度增加。
  - 单 SQLite 多家谱：实现简单，但隔离和迁移颗粒度不足。

### 决策 2：应用启动时扫描目录并构建 FamilyRegistry
- 结论：服务启动执行全量扫描，读取各库 `meta` 表并注册为可用家谱；扫描结果保存在内存注册表。
- 原因：
  - 与“文件即家谱”模型匹配，启动即可获得可用清单。
  - 便于在路由层按 `family_id` 做数据库选择。
- 备选方案：
  - 单独中心索引库：冷启动更快，但会引入一致性维护成本。
  - 惰性按需打开：启动快，但首请求时延不可控。

### 决策 3：每库独立迁移与容错
- 结论：扫描阶段按 `schema_version` 对每个家谱库执行迁移；单库失败仅标记不可用，不中断服务启动。
- 原因：
  - 避免“一个坏库拖垮全局”。
  - 支持不同家谱逐步升级。
- 备选方案：
  - 启动失败即中止：一致性强，但可用性差。

### 决策 4：家谱级 RBAC（三角色）
- 结论：角色固定为 `creator`、`admin`、`member`。
- 关键规则：
  - 每个家谱必须至少有 1 个 `creator`。
  - `admin` 不得降级或移除 `creator`。
  - 删除家谱和批量删除类高风险操作必须二次确认并写审计。
- 原因：
  - 角色简洁，便于产品和实现快速收敛。
  - 满足绝大多数家庭协作治理场景。
- 备选方案：
  - 增加 `viewer` 只读角色：更细粒度，但当前不是刚需。
  - 自定义 ACL：灵活，但实现与沟通成本显著上升。

### 决策 5：树图接口采用按层加载
- 结论：树查询接口必须支持 `root_person_id` 与 `depth`，并返回 `has_more`/`next_cursor` 等续取信号。
- 原因：
  - 控制大规模家谱一次性加载压力。
  - 与前端“展开分支”交互天然匹配。
- 备选方案：
  - 一次返回全树：实现简单，但性能和体验风险高。

### 决策 6：SQLite 并发策略标准化
- 结论：统一启用 `WAL`、`busy_timeout`，所有写操作使用短事务。
- 原因：
  - 在多用户低并发场景下可显著提升稳定性。
  - 降低 `database is locked` 概率。
- 备选方案：
  - 默认 journal 模式：冲突概率更高。
  - 强制应用层串行：简单但吞吐差。

## 风险 / 权衡

- [风险] 家谱目录文件数量增加后启动扫描耗时上升
  - 缓解措施：后续可引入增量扫描与 catalog 缓存。

- [风险] SQLite 在突发并发写入下存在锁等待
  - 缓解措施：短事务、重试策略、写操作串行化队列（按家谱维度）。

- [风险] 成员误操作导致重要数据删除
  - 缓解措施：高风险操作二次确认、审计日志、文件级备份与恢复流程。

- [权衡] 单家谱单文件简化运维，但跨家谱分析能力较弱
  - 缓解措施：后续通过离线导出聚合或分析副本解决。

## 迁移计划

1. 创建变更对应规范与任务，冻结接口与数据契约。
2. 先实现家谱文件扫描、注册与基础健康状态接口。
3. 实现核心数据模型（人物、关系、事件、附件）及校验规则。
4. 实现 RBAC 中间件与审计写入。
5. 实现树查询接口（按层加载）并与前端联调。
6. 补充迁移脚本与回滚机制（每库独立）。

回滚策略：
- 应用层：回退到上一版本服务。
- 数据层：按家谱文件恢复最近备份快照。
- 单库故障：标记该家谱不可用并触发告警，不阻断其他家谱。
